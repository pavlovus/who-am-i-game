# Технічна документація та план реалізації проєкту: Онлайн-гра «Хто я?» (FULL_CONTEXT)

Цей документ є єдиним джерелом правди (Single Source of Truth) для команди розробки. Тут детально описано архітектуру, мережевий протокол, взаємодію потоків та структуру бази даних для фінального проєкту з курсу "Розробка клієнт-серверних застосувань".

---

## 1. Огляд архітектури
- **Тип застосунку:** Клієнт-серверна архітектура, реалізована на чистій Java.
- **Сервер (Server):** Центральний вузол, який підтримує постійні TCP-з'єднання з клієнтами, керує станом ігрових кімнат, зберігає дані у БД, перевіряє бізнес-логіку та розсилає ігровий стан.
- **Клієнт (Client):** Десктопний Java-застосунок (Swing або JavaFX), що містить графічний інтерфейс для гравця. Завдяки постійному TCP-з'єднанню може миттєво отримувати оновлення від сервера.

**Обов'язкові компоненти (за ТЗ):**
1. Робота по мережі (TCP).
2. Шифрування (AES-128) та власний протокол.
3. Багатопоточність (Multithreading).
4. База даних (JDBC / PostgreSQL).

## 2. Мережевий рівень та Багатопоточність
### 2.1 Мережевий рівень
- **Протокол передачі:** TCP-сокети (`java.net.ServerSocket` та `java.net.Socket`).
- **Синхронізація (Real-time):** Завдяки постійному дуплексному з'єднанню сервер може **самостійно "пушити" (надсилати)** дані клієнту, як тільки інший гравець зробив хід. Ніякого Polling-у не потрібно.
- **Відновлення зв'язку:** Клієнт повинен вміти перехоплювати виключення (наприклад, `SocketException`) і намагатися перепідключитися.

### 2.2 Багатопоточна модель
- **Прийом з'єднань:** Головний потік сервера циклічно викликає `serverSocket.accept()`.
- **Обробка клієнтів:** Кожне підключення обслуговується у своєму власному потоці (через `ExecutorService`). Це дозволяє одночасно слухати вхідні пакети від усіх підключених клієнтів.
- **Ігрові кімнати:** Кожна кімната (`GameRoom`) є потокобезпечним об'єктом. Для уникнення стану гонитви використовуються:
  - `ConcurrentHashMap` для зберігання списку підключених клієнтів та кімнат.
  - `AtomicInteger` для лічильників питань і спроб.
  - `synchronized` блоки для критичних ділянок коду (наприклад, зміна статусу гри).

## 3. Власний бінарний протокол пакетів
Замість передачі тексту чи JSON, використовується власний бінарний протокол, який передається безпосередньо через потоки вводу/виводу TCP-сокета (`InputStream` / `OutputStream`).

### 3.1 Структура пакета (`Packet`)
Кожен пакет має суворий формат:
- **`magicByte` (1 байт):** Маркер початку пакета (наприклад, `0x13`).
- **`clientId` (4 байти):** Унікальний ідентифікатор відправника/отримувача.
- **`packetType` (4 байти):** Тип команди (Enum: `AUTH_REQUEST`, `GAME_START`, `QUESTION` тощо).
- **`payloadLength` (4 байти):** Довжина масиву даних.
- **`crc16` (2 байти):** Контрольна сума для перевірки цілісності.
- **`payload` (N байт):** Зашифроване або відкрите корисне навантаження.

### 3.2 Шифрування (Cryptography)
- **Алгоритм:** Симетричне шифрування AES-128.
- **Застосування:** Дані у `payload` (зокрема ім'я загаданого персонажа, повідомлення чату/питання) шифруються перед відправкою.

## 4. Безпека та Авторизація (Production-ready)
- **Хешування паролів:** Використовуватиметься безпечний алгоритм хешування паролів **Bcrypt** або **Argon2**. Паролі ніколи не зберігаються у відкритому вигляді.
- **Сесії та JWT:** Після успішної авторизації сервер генерує безпечні JWT токени (через `io.jsonwebtoken.jjwt`). 
  - Надалі клієнт може передавати цей токен у пакеті `AUTH_RESTORE` або в заголовках `payload` при перепідключенні, щоб не вводити логін/пароль знову.
  - Реалізується механізм Access (короткоживучі) та Refresh (довгоживучі) токенів.

## 5. База даних (Database)
- **СКБД:** **PostgreSQL** — надійна, масштабована реляційна база даних.
- **Доступ:** JDBC (з драйвером `org.postgresql:postgresql`) з обов'язковим використанням `PreparedStatement` для захисту від SQL-ін'єкцій. Буде використовуватись пул з'єднань HikariCP.

### 5.1 Схема таблиць
**`users`**
- `id` (SERIAL PRIMARY KEY)
- `login` (VARCHAR(50) UNIQUE NOT NULL)
- `password_hash` (VARCHAR(255) NOT NULL)
- `role` (VARCHAR(20) DEFAULT 'PLAYER')
- `wins` (INTEGER DEFAULT 0)
- `losses` (INTEGER DEFAULT 0)
- `avg_questions` (DOUBLE PRECISION DEFAULT 0.0)
- `is_blocked` (BOOLEAN DEFAULT FALSE)

**`characters`**
- `id` (SERIAL PRIMARY KEY)
- `name` (VARCHAR(100) NOT NULL)
- `category` (VARCHAR(50) NOT NULL)
- `is_approved` (BOOLEAN DEFAULT FALSE)

**`games_history`**
- `id` (SERIAL PRIMARY KEY)
- `room_code` (VARCHAR(20) NOT NULL)
- `guesser_id` (INTEGER REFERENCES users(id))
- `riddler_id` (INTEGER REFERENCES users(id))
- `character_id` (INTEGER REFERENCES characters(id))
- `winner_id` (INTEGER REFERENCES users(id))
- `questions_asked` (INTEGER DEFAULT 0)
- `created_at` (TIMESTAMP DEFAULT CURRENT_TIMESTAMP)

## 6. Детальна структура проєкту
Проєкт складатиметься з двох основних модулів — Серверного та Клієнтського, а також спільного модуля для протоколу пакетів.

### Спільний модуль (Shared / Protocol)
```text
src/main/java/com/whoami/protocol/
├── packets/
│   ├── Packet.java               # Базова структура бінарного пакету
│   ├── PacketType.java           # Enum усіх типів команд
│   ├── PacketBuilder.java        # Utility-клас для збірки/розбору пакетів з байтів
│   └── PayloadSerializer.java    # Конвертер об'єктів у байти (JSON/Custom -> bytes)
├── crypto/
│   ├── AESEncrypter.java         # Клас для AES шифрування
│   ├── AESDecrypter.java         # Клас для AES розшифрування
│   ├── Crc16.java                # Алгоритм підрахунку CRC16
│   └── PasswordEncoder.java      # Обгортка над Bcrypt/Argon2
└── models/
    ├── GameState.java            # Модель стану гри
    └── UserProfile.java          # Скорочена модель користувача
```

### Серверний модуль (Server)
```text
src/main/java/com/whoami/server/
├── MainServer.java               # Точка входу сервера, запуск ServerSocket
├── network/
│   ├── ConnectionListener.java   # Приймає serverSocket.accept()
│   ├── ClientHandler.java        # Потік обробки конкретного клієнта (читає InputStream)
│   └── PacketRouter.java         # Перенаправлення вхідних пакетів до відповідних сервісів
├── auth/
│   ├── AuthService.java          # Логіка логіну/реєстрації
│   └── JwtProvider.java          # Генерація та валідація Access/Refresh токенів
├── game/
│   ├── RoomManager.java          # Створення кімнат, метчмейкінг
│   ├── GameRoom.java             # Окремий суб'єкт гри (стан, черга ходів)
│   └── LogicController.java      # Перевірка правильності відповідей/логіки гри
├── database/
│   ├── ConnectionPool.java       # HikariCP пул з'єднань PostgreSQL
│   ├── UserDAO.java              # CRUD для users
│   ├── CharacterDAO.java         # CRUD для characters
│   └── GameDAO.java              # Запис історії ігор
└── admin/
    ├── AdminGuiController.java   # Контролер інтерфейсу адміна
    └── views/                    # Класи UI для серверної панелі
```

### Клієнтський модуль (Client)
```text
src/main/java/com/whoami/client/
├── MainClient.java               # Точка входу клієнта
├── network/
│   ├── ServerConnection.java     # Управління TCP-сокетом, підключення/реконнект
│   ├── IncomingPacketTask.java   # Окремий потік прослуховування пакетів від сервера
│   └── OutgoingQueue.java        # Черга пакетів на відправку на сервер
├── state/
│   ├── ClientContext.java        # Зберігає JWT, поточного юзера, поточну кімнату
│   └── AppStateManager.java      # Навігатор між екранами
└── ui/
    ├── controllers/              # Обробники подій кнопок
    └── views/
        ├── LoginScreen.java      # Екран входу та реєстрації
        ├── MainMenuScreen.java   # Створення/пошук кімнати
        ├── GameScreen.java       # Основний інтерфейс гри (чат, кнопки дій)
        └── PostGameScreen.java   # Результати та статистика
```
